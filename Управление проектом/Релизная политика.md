## Релизная политика
> Релизная политика - это план, как выпускать обновления приложения "Goal Ways", чтобы оно работало стабильно и безопасно. 

### 1. Основные правила релизов

- **Как часто выпускать:**

Каждые 2–4 недели (спринты по 2 недели). Начинаем с MVP, потом добавляем фичи по релизах.

- **Версии:**
  
Используем номера, например, v1.0.0 (MVP), v1.1.0 (Release 2).

- **Безопасность:**

Данные шифруются, AI не делится ими без вашего согласия. Если ошибка (например, сбой в AI), откатываем версию за 24 часа.

- **Фидбек:**

После релиза спрашиваем пользователей (опросы), исправляем проблемы в следующем обновлении.

<br>

### 2. Тестирование:
2.1. Сначала тестируем внутри команды (альфа).

2.2. Тестируем на 100–500 пользователях (бета): данные не теряются, AI работает правильно, скорость (<5 сек загрузка), батарею (<15%) и безопасность (шифрование данных).

<br>

### 3. Шаги для каждого релиза (связь с DFD):

1. Планирование: Смотрим на USM (список историй пользователей) и DFD — какие данные нужны (например, для AI-разбивки цели).
   
2. Разработка: Добавляем фичи, проверяя поток данных (например, ввод → AI → хранение → вывод).
   
3. Тестирование: Убеждаемся, что данные не ломаются (например, цель разбивается правильно).
   
4. Выпуск: Публикуем в Google Play/Ru Store, уведомляем пользователей.
   
5. Мониторинг: Смотрим метрики (ошибки, использование), чиним баги.

<br>

### 4. Релизы по версиям:
1. MVP (v1.0.0): Базовые фичи (регистрация, цель, прогресс, достижения).
   
2. Release 2 (v1.1.0): Архив задач, синхронизация, уведомления, уровни. Добавляем хранение данных (архив) и AI-анализ.
   
3. Release 3 (v1.2.0): Сообщества, магазин, экспорт, челленджи, оффлайн. Проверяем поток для групп (данные между пользователями).
   
4. Release 4 (v2.0.0): Группы, сообщества. Масштабируем на больше пользователей, проверяя безопасность данных.
